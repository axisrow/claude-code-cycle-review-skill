# Cycle Review — Claude Code Plugin

> [English version](README.md)

Автоматический цикл ревью PR для Claude Code. При первом запуске спрашивает, какие ревью-боты у вас есть (`@claude`, `@codex` или оба). Планирует стратегию мерджа, когда открыто несколько PR, запрашивает ревью у каждого настроенного ревьюера, интеллектуально сортирует комментарии ревьюеров (ботов и людей), применяет исправления и повторяет до аппрува — затем squash-мержит.

## Установка

```bash
git clone https://github.com/axisrow/claude-code-cycle-review-skill.git
cp -r claude-code-cycle-review-skill/skills/cycle-review ~/.claude/skills/
cp -r claude-code-cycle-review-skill/commands ~/.claude/
```

### Требования

- [GitHub CLI](https://cli.github.com/) (`gh`), авторизованный в вашем аккаунте
- [`jq`](https://jqlang.github.io/jq/) — для чтения и записи конфига онбординга
- Claude Code с ревьюером через GitHub Actions (`claude[bot]`), Codex (`@codex`) или любой другой PR-ревьюер

### Настройка

Перед использованием скилла выполните команду один раз:

```
/install-github-app
```

Это устанавливает GitHub App и CI, чтобы Claude Code мог делать ревью и редактировать комментарии на pull requests.

## Использование

```
/cycle-review [pr-numbers...] [onboard]
/cr           [pr-numbers...] [onboard]
```

Обе формы эквивалентны — `/cr` короткий алиас.

Если номера PR не указаны, скилл определяет текущий PR из ветки и дополнительно собирает остальные ваши открытые PR для планирования стратегии мерджа. Номера принимаются в любом свободном формате (`/cr 20 21 25`, `/cr 20, 21, 25` и т.д.). Чужие PR никогда не подключаются автоматически — передайте их номера явно, если хотите включить в очередь.

## Онбординг ревьюеров

При **первом запуске** скилл спрашивает, какие ревью-боты у вас установлены: `@claude`, `@codex` или оба. Выбор сохраняется глобально (один раз на пользователя) в:

```
~/.claude/cycle-review/config.json
```

```json
{
  "reviewers": ["@claude", "@codex"],
  "version": 1
}
```

Дальше скилл молча использует этот список: пингует каждого настроенного ревьюера в шаге «Запрос ревью» и ждёт ответа от каждого из них. Чтобы изменить выбор позже — запустите короткую команду:

```
/cr onboard
```

Старые формы `--onboard` и `--reconfigure` тоже остаются совместимыми.

Файл глобальный (а не в репозитории), поэтому он не засоряет проверяемые проекты и переиспользуется во всех репозиториях.

## Как это работает

Скилл запускает автоматический цикл (онбординг + 8 шагов):

### 0. Онбординг ревьюеров (один раз)

При отсутствии конфига (или с аргументом `onboard`) спрашивает, какие боты доступны (`@claude` / `@codex` / оба), и сохраняет выбор в `~/.claude/cycle-review/config.json`. Если конфиг уже есть — шаг пропускается.

### 1. Multi-PR стратегия

Когда у вас открыто несколько PR, скилл строит карту пересечений по файлам и стекам, затем автономно объявляет порядок мерджа (от раннего при пересечениях; в любом порядке при независимости). Можно прервать и переопределить.

### 2. Запрос ревью

Пингует **всех настроенных ревьюеров одним комментарием** — тело начинается со всех настроенных упоминаний (`@claude @codex`, если включены оба, или одно), с просьбой сфокусироваться только на критичных проблемах (баги, безопасность, логические ошибки, потеря данных, производительность). Косметические замечания явно не приветствуются.

### 3. Ожидание ответа ревьюера

Ожидание выполняет **один унифицированный драйвер** (`skills/cycle-review/wait-for-reviews.sh`), а не самописный цикл, переизобретаемый каждый раз. На каждом тике он `sleep`-ит, перечитывает **все** комментарии **всех** ботов и определяет завершение по **телу** последнего комментария каждого бота — никогда по количеству комментариев.

- **Почему по телу, а не по счётчику**: Claude **всегда редактирует свой единственный комментарий на месте**, а не создаёт новый. Поэтому ожидание по счётчику («пока у бота не станет больше одного комментария») висит вечно — именно этот баг однажды застопорил PR. Проверка тела (`Claude finished`) корректно ловит правку на месте.
- **Идемпотентность по PR**: состояние ожидания сохраняется (ключ `owner/repo/PR`). Если цикл прервался — смена тура, компакция контекста, потеря фоновой задачи — повторный запуск той же команды **возобновляет** ожидание, а не стартует заново: уже завершившиеся ревьюеры не ждутся повторно, таймаут считается от исходного старта. Флаг `RESET=1` начинает новое ожидание для следующего раунда ревью.
- **Тайминги на ревьюера** (Codex медленнее, ~5 мин против ~2 мин): Claude таймаутится на 7 мин, Codex — на 12 мин. Интервал опроса 30с. Всё настраивается через env.
- **Завершение Codex**: появление итогового ревью/комментария вместо плейсхолдера прогресса.
- **Лимит Claude**: если ревью Claude падает из-за исчерпания лимита аккаунта, то — если Codex настроен — цикл отбрасывает Claude и продолжает только на Codex; если Codex не настроен, уведомляет вас, что лимит исчерпан, и останавливается (восстановления лимита он не ждёт).

Запускается в фоновом `Bash` с отключённым sandbox (GitHub API недоступен из sandbox); ведущий `sleep N && …` заблокирован, но `sleep` *внутри* цикла драйвера — нормально.

### 4. Анализ и сортировка комментариев

Собирает комментарии от **всех ревьюеров** (ботов и людей) — как issue-комментарии, так и PR review comments. Субагент сортирует каждый комментарий, читая реальный код:

| Вердикт | Действие |
|---|---|
| `FIX` | Применить исправление (только после проверки утверждения по реальному коду) |
| `ALREADY_FIXED` | Ответить с указанием коммита, где это уже исправлено |
| `SKIP` | Ответить, что это косметическое замечание |
| `IRRELEVANT` | Ответить, что комментарий не относится к PR |
| `CONFLICTING` | Процитировать противоречащий комментарий, попросить уточнение |
| `HALLUCINATION` | Ответить с доказательствами из кодовой базы, опровергающими утверждение |

Финализация происходит, когда не осталось вердиктов `FIX` — скилл не требует явного `APPROVED` review-state, поскольку bot-ревьюеры почти никогда его не выставляют.

### 5. Исправление

Применяет фиксы только для вердикта `FIX`. Запускает линтер и тесты перед продолжением.

### 6. Коммит и пуш

Conventional commit, пуш в remote, возврат к шагу 2.

### 7. Проверка CI перед мерджем

Использует `gh pr checks --watch` для ожидания завершения всех проверок. Если какая-то упала — читает логи неудачного прогона, исправляет причину, возвращается к шагу 6. Останавливается после 2 неудачных попыток на одной и той же проверке.

### 8. Завершение

При аппруве без оставшихся комментариев и зелёном CI — squash-мерж и удаление ветки. Если активна multi-PR очередь — пересчитывает пересечения и переходит к следующему PR.

## Ключевые особенности

- **Онбординг ревьюеров** — при первом запуске спрашивает `@claude` / `@codex` / оба, хранит выбор глобально в `~/.claude/cycle-review/config.json`, переназначается через `/cr onboard`
- **Унифицированное ожидание по телу** — один коммитимый драйвер перечитывает все комментарии ботов на каждом тике и определяет завершение по телу комментария, поэтому правки Claude на месте не вызывают зависания по счётчику
- **Возобновляемое ожидание** — состояние ожидания сохраняется по каждому PR, поэтому прерванный цикл продолжается с места остановки, а не стартует ожидание заново
- **Тайминги на ревьюера** — знает, что Codex медленнее Claude (~5 мин против ~2 мин), и ждёт каждого по своему расписанию
- **Фолбэк при лимите Claude** — если Claude упёрся в лимит, продолжает на Codex (если настроен), иначе уведомляет и останавливается без ожидания восстановления лимита
- **Multi-PR планирование** — обнаруживает пересечения файлов и PR-стеки, выбирает порядок мерджа
- **Sandbox-aware ожидание** — использует `gh run watch` и фоновый polling; не полагается на заблокированный `sleep N && ...`
- **Умный поллинг** — отслеживает комментарий ревьюера по ID вместо фиксированного ожидания
- **Мульти-ревьюер** — обрабатывает комментарии от всех ревьюеров, не только от бота
- **Верификация утверждений** — проверяет каждое утверждение ревьюера по реальной кодовой базе
- **Детекция галлюцинаций** — ловит несуществующие функции, неверные номера строк, ложные конвенции
- **Обработка противоречий** — обнаруживает конфликтующие запросы между циклами ревью
- **Пропуск косметики** — пропускает стилевые замечания с вежливым объяснением
- **Линтер и тесты** — запускает `ruff` и `pytest` перед каждым пушем
- **Гейт CI** — ждёт зелёный CI перед мерджем, автоматически разбирает падения
