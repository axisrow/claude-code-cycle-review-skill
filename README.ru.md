# Cycle Review — Claude Code Plugin

> [English version](README.md)

Автоматический цикл ревью PR для Claude Code с **двумя режимами ревью**:

- **Облачный** (по умолчанию) — пингует ваши GitHub-боты (`@claude`, `@codex` или оба), ждёт их ответа на PR, сортирует комментарии, исправляет и автономно доводит до CI и squash-мерджа.
- **Локальный** — ревью делает встроенный **субагент Claude**, читающий дифф PR напрямую (без пинга ботов и ожидания GitHub). Он записывает вердикты каждого раунда комментарием в PR, исправляет, коммитит и пушит — но **не мержит сам**: финальный merge вы запускаете вручную, когда готовы. Локальный режим — **только для Claude Code** (у Codex нет аналогичного локального плагина), поэтому Codex в нём не участвует.

При первом запуске скилл спрашивает, какие ревью-боты у вас есть и какой режим по умолчанию. Также планирует стратегию мерджа, когда открыто несколько PR, проверяет, что каждый PR реализует связанный issue на 100% до ревью, интеллектуально сортирует комментарии ревьюеров (ботов и людей), применяет исправления и повторяет до аппрува.

## Установка

```bash
git clone https://github.com/axisrow/claude-code-cycle-review-skill.git
cp -r claude-code-cycle-review-skill/skills/cycle-review ~/.claude/skills/
cp -r claude-code-cycle-review-skill/commands ~/.claude/
```

### Требования

- [GitHub CLI](https://cli.github.com/) (`gh`), авторизованный в вашем аккаунте
- [`jq`](https://jqlang.github.io/jq/) — для чтения и записи конфига онбординга
- Claude Code с ревьюером через GitHub Actions (`claude[bot]`), Codex (`@codex`) или любой другой PR-ревьюер

### Настройка

Перед использованием скилла выполните команду один раз:

```
/install-github-app
```

Это устанавливает GitHub App и CI, чтобы Claude Code мог делать ревью и редактировать комментарии на pull requests.

## Использование

```
/cycle-review [local|cloud] [pr-numbers...] [onboard]
/cr           [local|cloud] [pr-numbers...] [onboard]
```

Обе формы эквивалентны — `/cr` короткий алиас.

Если номера PR не указаны, скилл определяет текущий PR из ветки и дополнительно собирает остальные ваши открытые PR для планирования стратегии мерджа. Номера принимаются в любом свободном формате (`/cr 20 21 25`, `/cr 20, 21, 25` и т.д.). Чужие PR никогда не подключаются автоматически — передайте их номера явно, если хотите включить в очередь.

Ведущий токен `local` или `cloud` (а также `--local` / `--cloud`) форсит режим для текущего запуска, переопределяя сохранённый по умолчанию — например, `/cr local 58` ревьюит PR #58 локально, а `/cr 58` использует режим по умолчанию.

## Онбординг (ревьюеры + режим по умолчанию)

При **первом запуске** скилл спрашивает две вещи и сохраняет их глобально (один раз на пользователя):

1. **Какие ревью-боты у вас есть** — `@claude`, `@codex` или оба (используются облачным режимом).
2. **Режим ревью по умолчанию** — `cloud` (пинг GitHub-ботов, автономно до мерджа) или `local` (встроенный субагент Claude, не мержит сам).

```
~/.claude/cycle-review/config.json
```

```json
{
  "reviewers": ["@claude", "@codex"],
  "mode": "cloud",
  "version": 2
}
```

Сохранённый `mode` — лишь значение по умолчанию: флаг `local`/`cloud` всегда переопределяет его для конкретного запуска. Чтобы изменить любой из выборов позже — запустите:

```
/cr onboard
```

Старые формы `--onboard` и `--reconfigure` тоже остаются совместимыми. Уже существующий конфиг `version: 1` (без поля `mode`) остаётся валидным и трактуется как `cloud`, пока вы не пройдёте онбординг заново.

Файл глобальный (а не в репозитории), поэтому он не засоряет проверяемые проекты и переиспользуется во всех репозиториях.

## Как это работает

Скилл запускает автоматический цикл (онбординг + 8 шагов):

### 0. Онбординг ревьюеров (один раз)

При отсутствии конфига (или с аргументом `onboard`) спрашивает, какие боты доступны (`@claude` / `@codex` / оба), и сохраняет выбор в `~/.claude/cycle-review/config.json`. Если конфиг уже есть — шаг пропускается.

### 1. Multi-PR стратегия

Когда у вас открыто несколько PR, скилл строит карту пересечений по файлам и стекам, затем автономно объявляет порядок мерджа (от раннего при пересечениях; в любом порядке при независимости). Можно прервать и переопределить.

### 1.5. Проверка полноты реализации issue (до ревью)

Перед пингом ботов скилл проверяет, что каждый PR реально реализует весь дизайн связанного issue — а не просто что код компилируется. Он читает связанный issue (`Closes #N`), превращает дизайн в чеклист и убеждается, что каждый пункт (формат вывода, флаг, маркер, граничный случай) присутствует в диффе. Любой пробел закрывается test-first и пушится **до** старта ревью.

Почему заранее: ревью-боты оценивают корректность *кода*, а не его *полноту* относительно issue — оба бота могут заапрувить PR, реализующий лишь половину дизайна. Проверка здесь экономит раунд ревью (и не даёт смержить недоделанный issue). Если пробел большой или дизайн неоднозначен, скилл показывает недостающие пункты и спрашивает, как поступить.

### 2. Запрос ревью

**Облачный:** пингует **всех настроенных ревьюеров одним комментарием** — тело начинается со всех настроенных упоминаний (`@claude @codex`, если включены оба, или одно), с просьбой сфокусироваться только на критичных проблемах (баги, безопасность, логические ошибки, потеря данных, производительность). Косметические замечания явно не приветствуются.

**Локальный:** спавнит встроенный **субагент Claude** (Agent tool), который читает `gh pr diff` и текущее содержимое затронутых файлов, ревьюит по той же планке «только критичное», проверяет каждое утверждение по реальному коду и возвращает структурированные находки. Бот не пингуется, ожидания GitHub нет — шаг 3 пропускается, находки идут сразу в сортировку (шаг 4).

### 3. Ожидание ответа ревьюера *(только облачный режим)*

Ожидание выполняет **один небольшой коммитимый драйвер** (`skills/cycle-review/wait-for-reviews.sh`, ~50 строк), а не самописный цикл, переизобретаемый каждый раз. Это намеренно «тупой» ожидатель: он просто `sleep`-ит одно фиксированное окно (`WAIT`, по умолчанию 300с — покрывает ~5 мин Codex и ~2 мин Claude), затем проверяет, доступен ли GitHub API, и печатает ровно одну строку:

- `DONE` — окно истекло, API доступен. Дальше сортировка (шаг 4) сама читает все комментарии и решает, что произошло.
- `ERROR` — комментарии PR не удалось прочитать после нескольких попыток (устойчивый сбой API — протухшая авторизация, rate limit, сеть). Это **не** «нет замечаний»: цикл останавливается, и вам предлагают проверить, например, `gh auth status`. Сбою никогда не дают притвориться аппрувом.

Он **не** определяет, кто закончил, не парсит тела комментариев, не отслеживает состояние по ревьюерам и не возобновляется — нет файла состояния и нечего сбрасывать. **Вся** интерпретация (кто ответил, релевантность, галлюцинации, сообщение о лимите Claude) происходит в сортировке на шаге 4, которая сама читает все три поверхности комментариев. Каждый новый раунд просто постит свежий запрос (шаг 2) и снова запускает ожидатель; если фоновый прогон потерян, перезапуск безвреден. Это пришло на смену переусложнённой версии на ~290 строк, чья машинерия определения завершения/возобновления плодила собственные баги (например, ожидание по счётчику однажды зависло навсегда, потому что Claude редактирует свой единственный комментарий на месте, а не создаёт новый).

Запускается в фоновом `Bash` с отключённым sandbox (GitHub API недоступен из sandbox); ведущий `sleep N && …` заблокирован, но `sleep` *внутри* цикла драйвера — нормально.

### 4. Анализ и сортировка комментариев

Собирает комментарии от **всех ревьюеров** (ботов и людей) — как issue-комментарии, так и PR review comments. Субагент сортирует каждый комментарий, читая реальный код:

| Вердикт | Действие |
|---|---|
| `FIX` | Применить исправление (только после проверки утверждения по реальному коду) |
| `ALREADY_FIXED` | Ответить с указанием коммита, где это уже исправлено |
| `SKIP` | Ответить, что это косметическое замечание |
| `IRRELEVANT` | Ответить, что комментарий не относится к PR |
| `CONFLICTING` | Процитировать противоречащий комментарий, попросить уточнение |
| `HALLUCINATION` | Ответить с доказательствами из кодовой базы, опровергающими утверждение |

В **локальном** режиме у находок субагента нет GitHub-комментария, на который можно ответить, поэтому каждый раунд скилл постит **один сводный комментарий** с вердиктами в PR — локальное ревью является ревьюером-источником истины.

Когда не осталось вердиктов `FIX` — это **финальный цикл**; скилл не требует явного `APPROVED` review-state, поскольку bot-ревьюеры почти никогда его не выставляют. Он оставляет ответы (облако) или сводку (локально) на не-`FIX` комментарии и переходит к финальной зачистке (шаг 6.5). После зачистки **облачный** режим идёт в CI и мерж; **локальный** останавливается и отчитывается — сам не мержит.

**Жёсткий лимит — 3 цикла.** Цикл — это один раунд «запрос ревью → сортировка → исправление». Если на 3-м цикле всё ещё есть вердикты `FIX`, скилл останавливается, а не уходит на 4-й круг — три раунда с открытыми замечаниями означают, что PR не сходится. Он передаёт управление вам со сводкой оставшихся замечаний и предложением сузить скоуп: вынести часть out of scope в отдельный issue/PR, чтобы основное изменение можно было смержить, либо переосмыслить подход. Лимит срабатывает только когда замечания не кончаются; чистый 1-й или 2-й раунд финализируется как обычно.

### 5. Исправление

Применяет фиксы только для вердикта `FIX`. Запускает линтер и тесты перед продолжением.

### 6. Коммит и пуш

Conventional commit, пуш в remote, возврат к шагу 2.

### 6.5. Финальная зачистка (последний цикл)

Достигается только на финальном цикле — когда в раунде нет вердиктов `FIX`. Поскольку ревью-раунда после него уже не будет, скилл за один проход применяет **все мелкие замечания, отложенные за все предыдущие раунды** (а не только за последний): каждый настоящий `SKIP` (косметика/стиль/нейминг) плюс любые разумные nice-to-have предложения ревьюеров. Исключаются вердикты, где нечего чинить (`HALLUCINATION` / `IRRELEVANT` / `CONFLICTING` / `ALREADY_FIXED`), дедуплицируются повторявшиеся между раундами замечания, прогоняются линт и тесты, коммитится зачистка — и сразу к CI, **без** запроса нового ревью. Если PR не накопил мелких замечаний — шаг становится no-op.

### 7. Проверка CI перед мерджем *(только облачный режим)*

Использует `gh pr checks --watch` для ожидания завершения всех проверок. Если какая-то упала — читает логи неудачного прогона, исправляет причину, коммитит фикс и перепроверяет (возврат к этому же шагу). Останавливается после 2 неудачных попыток на одной и той же проверке. Локальный режим до этого шага не доходит.

### 8. Завершение *(только облачный режим)*

**Облачный:** при аппруве без оставшихся комментариев и зелёном CI — squash-мерж и удаление ветки. Если активна multi-PR очередь — пересчитывает пересечения и переходит к следующему PR.

**Локальный:** **не мержит** — он review-only на мерже по дизайну. После зачистки останавливается и отчитывается о том, что исправлено; финальный merge вы запускаете сами, когда готовы.

## Ключевые особенности

- **Два режима ревью** — `cloud` (GitHub-боты, автономно до мерджа) и `local` (встроенный субагент Claude, не мержит сам; только для Claude Code). Сохраняется как режим по умолчанию в конфиге и переопределяется для запуска ведущим флагом `local`/`cloud`
- **Онбординг ревьюеров** — при первом запуске спрашивает `@claude` / `@codex` / оба плюс режим по умолчанию, хранит выбор глобально в `~/.claude/cycle-review/config.json`, переназначается через `/cr onboard`
- **Простое ожидание фиксированным окном (облако)** — один драйвер на ~50 строк ждёт одно окно, затем проверяет API и печатает только `DONE` / `ERROR`; он не хранит состояние по ревьюерам и не возобновляется — вся интерпретация живёт в сортировке. Сбой (`ERROR`) никогда не путается с «нет замечаний»
- **Фолбэк при лимите Claude** — если Claude упёрся в лимит, продолжает на Codex (если настроен), иначе уведомляет и останавливается без ожидания восстановления лимита
- **Multi-PR планирование** — обнаруживает пересечения файлов и PR-стеки, выбирает порядок мерджа
- **Sandbox-aware ожидание** — запускает ожидатель и вызовы `gh` в фоне с отключённым sandbox; не полагается на заблокированный ведущий `sleep N && ...`
- **Мульти-ревьюер** — обрабатывает комментарии от всех ревьюеров, не только от бота
- **Верификация утверждений** — проверяет каждое утверждение ревьюера по реальной кодовой базе
- **Детекция галлюцинаций** — ловит несуществующие функции, неверные номера строк, ложные конвенции
- **Обработка противоречий** — обнаруживает конфликтующие запросы между циклами ревью
- **Пропуск косметики** — пропускает стилевые замечания с вежливым объяснением
- **Линтер и тесты** — запускает `ruff` и `pytest` перед каждым пушем
- **Гейт CI** — ждёт зелёный CI перед мерджем, автоматически разбирает падения
