# Cycle Review — Claude Code Plugin

> [English version](README.md)

Автоматический цикл ревью PR для Claude Code. Планирует стратегию мерджа, когда открыто несколько PR, запрашивает ревью, интеллектуально сортирует комментарии ревьюеров (ботов и людей), применяет исправления и повторяет до аппрува — затем squash-мержит.

## Установка

```bash
git clone https://github.com/axisrow/claude-code-cycle-review-skill.git
cp -r claude-code-cycle-review-skill/skills/cycle-review ~/.claude/skills/
cp -r claude-code-cycle-review-skill/commands ~/.claude/
```

### Требования

- [GitHub CLI](https://cli.github.com/) (`gh`), авторизованный в вашем аккаунте
- Claude Code с ревьюером через GitHub Actions (`claude[bot]`) или любой другой PR-ревьюер

### Настройка

Перед использованием скилла выполните команду один раз:

```
/install-github-app
```

Это устанавливает GitHub App и CI, чтобы Claude Code мог делать ревью и редактировать комментарии на pull requests.

## Использование

```
/cycle-review [pr-numbers...]
/cr           [pr-numbers...]
```

Обе формы эквивалентны — `/cr` короткий алиас.

Если номера PR не указаны, скилл определяет текущий PR из ветки и дополнительно собирает остальные ваши открытые PR для планирования стратегии мерджа. Номера принимаются в любом свободном формате (`/cr 20 21 25`, `/cr 20, 21, 25` и т.д.). Чужие PR никогда не подключаются автоматически — передайте их номера явно, если хотите включить в очередь.

## Как это работает

Скилл запускает автоматический цикл из 8 шагов:

### 1. Multi-PR стратегия

Когда у вас открыто несколько PR, скилл строит карту пересечений по файлам и стекам, затем автономно объявляет порядок мерджа (от раннего при пересечениях; в любом порядке при независимости). Можно прервать и переопределить.

### 2. Запрос ревью

Оставляет комментарий с просьбой сфокусироваться только на критичных проблемах (баги, безопасность, логические ошибки, потеря данных, производительность). Косметические замечания явно не приветствуются.

### 3. Ожидание ответа ревьюера

Sandbox-aware ожидание:
- **Основное**: `gh run watch` — нативный блокирующий watch на workflow Claude Action, самописный цикл не нужен.
- **Fallback**: poll комментария по ID — находит последний коммент `claude[bot]`, опрашивает каждые 30с на маркер `Claude finished`. Выполняется в фоне, чтобы не упираться в заблокированный `sleep N && ...`.
- Таймаут: 7 минут с момента создания комментария.

### 4. Анализ и сортировка комментариев

Собирает комментарии от **всех ревьюеров** (ботов и людей) — как issue-комментарии, так и PR review comments. Субагент сортирует каждый комментарий, читая реальный код:

| Вердикт | Действие |
|---|---|
| `FIX` | Применить исправление (только после проверки утверждения по реальному коду) |
| `ALREADY_FIXED` | Ответить с указанием коммита, где это уже исправлено |
| `SKIP` | Ответить, что это косметическое замечание |
| `IRRELEVANT` | Ответить, что комментарий не относится к PR |
| `CONFLICTING` | Процитировать противоречащий комментарий, попросить уточнение |
| `HALLUCINATION` | Ответить с доказательствами из кодовой базы, опровергающими утверждение |

Финализация происходит, когда не осталось вердиктов `FIX` — скилл не требует явного `APPROVED` review-state, поскольку bot-ревьюеры почти никогда его не выставляют.

### 5. Исправление

Применяет фиксы только для вердикта `FIX`. Запускает линтер и тесты перед продолжением.

### 6. Коммит и пуш

Conventional commit, пуш в remote, возврат к шагу 2.

### 7. Проверка CI перед мерджем

Использует `gh pr checks --watch` для ожидания завершения всех проверок. Если какая-то упала — читает логи неудачного прогона, исправляет причину, возвращается к шагу 6. Останавливается после 2 неудачных попыток на одной и той же проверке.

### 8. Завершение

При аппруве без оставшихся комментариев и зелёном CI — squash-мерж и удаление ветки. Если активна multi-PR очередь — пересчитывает пересечения и переходит к следующему PR.

## Ключевые особенности

- **Multi-PR планирование** — обнаруживает пересечения файлов и PR-стеки, выбирает порядок мерджа
- **Sandbox-aware ожидание** — использует `gh run watch` и фоновый polling; не полагается на заблокированный `sleep N && ...`
- **Умный поллинг** — отслеживает комментарий ревьюера по ID вместо фиксированного ожидания
- **Мульти-ревьюер** — обрабатывает комментарии от всех ревьюеров, не только от бота
- **Верификация утверждений** — проверяет каждое утверждение ревьюера по реальной кодовой базе
- **Детекция галлюцинаций** — ловит несуществующие функции, неверные номера строк, ложные конвенции
- **Обработка противоречий** — обнаруживает конфликтующие запросы между циклами ревью
- **Пропуск косметики** — пропускает стилевые замечания с вежливым объяснением
- **Линтер и тесты** — запускает `ruff` и `pytest` перед каждым пушем
- **Гейт CI** — ждёт зелёный CI перед мерджем, автоматически разбирает падения
